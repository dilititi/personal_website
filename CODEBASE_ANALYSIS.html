1 · Overview — the reframing that explains the mess

This is not a one-off personal site. It is a reusable bilingual (EN/ZH) portfolio template with a full in-browser content & style editor, an AI auto-fill importer, and a static, no-backend publishing model. “Chen, film student in Hangzhou” is demo seed data.

Evidence: README.md clones github.com/dilititi/personal_website; CONTENT_GUIDE.md is “a manual for users of this template”; the headline feature is AI auto-fill (ImportPanel.jsx). Most of the friction below is the seam between two generations of the project — a simple hand-edited site that grew into a templated product.

2 · Recurring themes

One data spine; config-driven everything

All copy lives in data.js as L(en, zh) pairs; sections render from a MODULES config sorted by order (App.jsx, NavShell.jsx). The site is data first, components second.

Two parallel “localStorage override + deep-merge” engines

  • Content (data-context.jsx) and style (style-context.jsx) duplicate the same recursive deep-merge (mergeSection ≡ mergeStyle) and the same load/save/lastSaved/mountedRef dance — down to the identical “Browser storage is full” message.

Pervasive defensive + bilingual rendering

426 occurrences of t() / ?. / === 'zh' / onError across all 19 components. Missing images hide, unknown media types get humanized fallbacks, every value resolves through pick(). This is a direct consequence of the data being user-editable.

Heavy investment in “edit-in-browser → promote-to-code”

Schema-driven validation (validation.js), export warnings for embedded base64 and missing public assets (export.js), AI import with markdown-fence stripping and one-click undo snapshots (ImportPanel.jsx). The tooling around the data is larger than the site itself.

Asymmetric bilingual discipline

The public site is rigorously EN/ZH; the workshop (editor labels, confirm dialogs, code comments, schema.js) is zh-first or zh-only — the template is authored by a Chinese developer for primarily Chinese users, but produces bilingual output sites.

3 · Evolution of thinking (oldest → newest)

The repository is legible as stratigraphy — each layer is a prior design still visible under the current one.

  • Workflow: README.md still says “edit data.js directly” and lists a flat, pre-editor file tree. CONTENT_GUIDE.md says the opposite: “all data is edited through the in-site editor; you don’t need to touch code.”
  • Module config: booleans → rich objects. data.js notes “legacy true/false overrides are still accepted… normalized,” and normalizeModuleConfig still handles the boolean case.
  • Persistence: per-feature keys → one unified store. Live migration shims read (and delete) chen.readingLog.userEntries (Library.jsx) and chen.photos.userEntries (Photography.jsx) into today’s single chen.content.overrides.
  • Content model: hardcoded → seed + user split. READING_LOG holds 24 curated entries; USER_READING_LOG = [] is the end-user’s; Library shows them as “curated” vs “personal.”
  • Styling: a fixed dark “ember” palette → a configurable engine (8 presets, ~60 derived vars). The old CSS var names are frozen fossils whose values inverted — --ink-void is now light cream #ebe2c8, --cream is near-black text — and are used 544× so they can’t be renamed.
  • Even the docs lag the code: CLAUDE.md says “only default layout is stable,” but compact/feature are fully implemented in sections.css.

4 · Contradictions & tensions

  • Docs disagree on the core workflow: README.md (edit code) vs CONTENT_GUIDE.md (never edit code) — same task, opposite instructions.
  • Two serializers, divergent output: per-section “Copy” emitted idiomatic L('en','zh') via export.js; “Copy All” and the download used data-context.exportResolvedData, plain JSON.stringify. The same editor pasted two different code styles into data.js. (Fixed — see §5.1.)
  • Sibling providers built differently: StyleProvider memoizes its value; DataProvider rebuilds resolvedData and value every render (baseData is memoized, resolvedData right below it is not). The two editors even default to opposite modes (content = modal, style = side).
  • Editable knobs wired to nothing: color.temperature, typography.personality, culture, style-mood, and anchors were persisted and had editor UI — including a full anchor-builder — but deriveStyleVars never read them. (design.alignment and motion.mode are live.) (Addressed — see §5.2.)
  • A polished editor whose persistence barely exists in prod: image/audio/PDF uploads POST to /api/upload, a dev-only Vite middleware (vite.config.js); in production those buttons silently no-op, and edits stay in localStorage until manually pasted into data.js.
  • Hand-maintained cross-file contracts: “medium must match Works.jsx’s WORK_MEDIA” (schema.js), “series must match PHOTO_SERIES”, plus two section lists (SECTION_KEYS vs EXPORTABLE_SECTIONS) — documented in prose/labels, not enforced. Softened by graceful fallbacks.

5 · Action plan (all 8 items)

Items 1 and 2 are already implemented in this pass; 3–8 are scoped for follow-up, ordered by impact-to-effort.

5.1 Unify the export path Done

Root cause: a second, dumber serializer in the data layer. Now there is one.

Changed: ContentEditor.jsx — “Copy All” and the data.js download now build their payload from EXPORTABLE_SECTIONS.map(s => exportLine(s.key, data[s.key])), matching the per-section Copy button. data-context.jsx — removed exportResolvedData() and its JSON.stringify-based jsLiteral. export.js is now the single serializer.

Result: every export button emits the same idiomatic L('en','zh') code. Note: SECTION_KEYS is now unused (left exported) — fold into item 6.

5.2 Resolve the dead style knobs Done

Stop the Style panel promising controls that render nothing; make the ones with a clean mapping real.

Changed: style-engine.js — color.temperature now tints background/surface warm (amber) or cool (blue), subtly (≤6% mix); added PERSONALITY_PAIRINGS so typography.personality selects a default display/body font pairing (explicit Display/Body still override → no preset regression). style.js + StyleEditor.jsx — culture/mood/anchors relabeled as a descriptive “Mood board” that saves with the style but does not drive CSS.

5.3 Close the production upload gap Planned

Highimpact

The editor’s image/audio/PDF buttons fail silently in production; users won’t know why nothing saves.

  1. In the file fields (editor/fields/FileField.jsx), branch on import.meta.env.PROD.
  2. In prod, disable the upload button and show a short hint (“dev-only — run npm run dev to write to public/”) or fall back to a manual path input.
  3. Add one line to CONTENT_GUIDE.md stating uploads only persist under vite dev.

5.4 Reconcile the documentation Planned

Highimpact, low effort

The README actively misleads (wrong workflow, outdated file tree); CLAUDE.md understates the code.

  1. Rewrite/retire README.md: point to the in-site editor workflow and regenerate the project-structure block (it omits data-context, style*, editor/, np-context, the styles/ split, etc.).
  2. Fix CLAUDE.md: remove “only default is stable” (compact/feature are implemented); note the single export path from item 1.

5.5 De-duplicate the provider / persistence machinery Planned

Medimpact

Two near-identical engines drift apart over time (they already have).

  1. Extract one shared deepMerge and a useLocalStorageState-style helper (load/save/lastSaved/first-render skip) used by both contexts.
  2. Memoize DataProvider’s resolvedData and context value to match StyleProvider.
  3. Drop the resetAll/resetData alias (keep one name).

5.6 Single source of truth for sections Planned

Medimpact, low effort

Two hardcoded section lists can silently diverge; item 1 already left one unused.

  1. Derive SECTION_KEYS from EXPORTABLE_SECTIONS (or remove it entirely now that exportResolvedData is gone).
  2. Make schema.js the canonical list that the data layer imports.

5.7 Add a minimal test net Planned

Medimpact

No test/lint/formatter exists, yet the riskiest, most-evolved code is exactly the merge/normalize/migration/export logic.

  1. Add Vitest.
  2. Cover mergeSection/mergeStyle, normalizeModuleConfig (boolean → object), the legacy migration shims, and an export.js jsLiteral round-trip (serialize → eval → deep-equal).

5.8 Sunset the legacy migration shims Planned

Lowimpact

Cleanup once users have migrated off the old per-feature keys.

  1. Remove the reads of chen.readingLog.userEntries (Library.jsx) and chen.photos.userEntries (Photography.jsx), or keep them behind a dated comment with a removal target.
