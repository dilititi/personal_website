# PRODUCT-DIRECTION · 产品方向评估与优先级

> 配套文档：`PLAN.md`（分阶段路线图）/ `SPEC-3-PERSIST.md`（持久化 / GitHub 发布）/ `ENGINEERING.md`（工程契约）/ `CLAUDE.md`（架构）。
> 状态：**方向已确认；P0–P2 与首轮反馈 UX 已实现，待任务制复测**（2026-06-12）。读者：项目所有者 + AI 编码 Agent。
> 范围：对「AI 辅助建站 · 5 步设计+部署 · 双模式」提案的独立评价与落地优先级。**这是决策依据,不是已承诺的路线**；要落地时按 `ENGINEERING.md` 出 spec。

---

## 1. 总体判断（先说结论）

- **方向成立**：「AI 辅助 + 可编辑 + 部署前护栏」的定位是对的,且契合现有这套有审美的双语静态模板。
- **但提案停在产品逻辑层**；一旦接到本仓库的真实约束（纯静态、无后端、内容在 `data.js`/localStorage、Render 托管、Phase 3 用浏览器 GitHub PAT，见 `SPEC-3-PERSIST.md`），有一半功能要重排或砍掉。
- 三条核心判断（§3）：
  - **3A** 你的资产是「取舍/审美」——让 AI **往固定骨架里填**,别让它生成骨架。
  - **3B** 纯静态架构否决了后端类功能；**Prompt Builder 是唯一不自毁的 AI 接入方式**。
  - **3C** 真正的前置阻塞是「模板与示例数据解耦」,不是「做 Wizard」。

---

## 2. 定位（是什么 / 不是什么）

- **是**：给一套有审美的双语静态模板,加一层「AI 辅助填充 + 排版护栏 + 透明导出/部署」。
- **不是**：通用 AI 建站平台（Framer AI / v0 / Lovable / Wix ADI 那类）。**不要在广度上对线**——那是一个有资金的拥挤赛道,单人/学生项目对线必输。
- **一句话定位**：_AI 辅助的可编辑作品集模板编辑器_——**生成是进入可编辑系统的起点,不是终点**。

---

## 3. 三个核心判断

### 3A · 资产是「取舍」：让 AI 填骨架,而非生成骨架

本仓库真正值钱的,是手工调过的 Landing masthead、reel 胶片、cinema 卡片、atlas 地图、以及 8 套 `STYLE_PRESETS`——这是一份*有品味的取舍*。改成「空白 prompt 生成任意结构」会把这份取舍丢掉,还继承所有 AI 建站工具的通病：生成炫、千篇一律、一改就崩。

- **最强的一步**：Goal Picker → 精选 preset（甚至零 AI 依赖也成立）。
- **最弱的一步**：空白 prompt 凭空生成结构 → 降级为「可选入口」,不做核心。
- 一句话：卖点是**「用户做不丑」**,不是「用户什么都能生成」。

### 3B · 纯静态架构否决一半功能；Prompt Builder 是唯一不自毁的接入

约束链：纯静态 → 无长期后端（`ENGINEERING.md` INV-8 / `check:dist`）→ 不能持有密钥、不能跑服务端推理 / 视觉模型。由此：

- 「轻后端代理」= 破坏无后端身份,等于换一个项目,**不是加功能**。
- 「参考图自动提取风格」需要视觉模型 = 必须 API/后端。**静态友好替代**：用 Prompt Builder 生成 prompt,让用户拿去外部带视觉的 Claude/GPT,输出 `StyleEditor` 的 JSON,再粘回来 → 「参考图」其实也是 Prompt Builder 的一种,不是站内视觉管线。
- 「截图坐标定位元素」在 iframe/预渲染环境又脆又贵；「传截图 + 文字描述 → 生成修改 prompt」用 10% 工作量拿 80% 价值。

所以 Prompt Builder 不只是「最容易」,而是**这套架构下唯一不破坏自身身份的 AI 接入**——而且它正是本项目所有者与 AI 协作的真实工作流,契合度极高。详见 §6 决策表。

### 3C · 真正前置阻塞：模板 / 示例数据解耦

`PLAN.md` Phase 4 里那条不起眼的「模板上手 / 示例数据解耦」,才是整个 AI 愿景的**结构性前提**。原因：Wizard 本质上是「按顺序往现有 override + style 状态里写」的友好前端（见 §5）,它依赖模板能干净地**空载实例化**（不含 Chen / 现有 demo 的硬编码内容）。没有这步,Wizard 无从谈起。

产品层面的硬话：**分发 / 有没有人用,是提案没解决的问题。** 一个很棒但没人用的生成器是个爱好。投入做 Wizard 之前,最便宜的验证是把现有这套已经很好看的模板做成「别人能 5 分钟用起来」,找到真实用户。

---

## 4. 对原 5 步流程的评价与重排

**原提案**：`0 preset/prompt 雏形 → 1 接大模型(iframe?) → 3 截图局部 → 2 参考图风格 → 4 高级动效`。

**评价**：产品方向对,但**顺序乱 + 技术方式发散**。重排为真实设计流程：

```
目标 → 雏形(含 AI 文案) → 风格 → 一键审计 → 部署
```

- 「参考图确定风格」必须早于「截图局部修改」（风格是系统,局部是细节）。
- **把「截图局部修改」移出向导**：抠局部视觉是天然迭代、进阶的事,塞进给小白的线性向导只会加摩擦。它属于 Mode B（手动）。
- 「接大模型用 iframe」是错的（§6）。

---

## 5. 双模式 + 关键接缝

| 模式                      | 面向     | 流程                                                                               |
| ------------------------- | -------- | ---------------------------------------------------------------------------------- |
| **A · 5 步设计+部署**     | 普通用户 | 目标 → 生成雏形 → 选风格 → 一键审计 → 部署                                         |
| **B · 手动细微设计+部署** | 进阶用户 | `ContentEditor` 精修 → `StyleEditor` 精修 → Layout 审计 → 受控 variant → 导出/部署 |

**关键洞见（两份外部评审都没点破）**：Wizard 与两个编辑器**写的是同一份数据**（`chen.content.overrides` + style config）。所以 Wizard 不是「另造一个生成器」,而是「按顺序写现有状态」的前端。

> 含义：双模式不是两套割裂系统,而是同一可编辑内核的两个入口。这是本项目相对「从零做 AI 平台」的**结构性成本优势**——生成不是黑箱,生成完即可无缝进入精修,避免了通用 AI 工具「生成炫、难改、一改就崩、导出不透明」的通病。

## 6. AI 接入选型（决策表）

| 方案                          | 机制                                         | 后端   | 破坏静态身份? | 结论                                                               |
| ----------------------------- | -------------------------------------------- | ------ | ------------- | ------------------------------------------------------------------ |
| iframe 嵌入 ChatGPT/Claude    | 把对话页嵌进编辑器                           | —      | —             | **✗** 登录/CORS/CSP 不可控,且**读不到你的应用状态、无法回填**,白嵌 |
| 复制 Prompt 到外部            | 一键复制 prompt 去外部模型                   | 无     | 否            | ✓ 可作 MVP 的一部分                                                |
| 用户自带 API key（前端直调）  | 浏览器用用户的 key 调 API                    | 无     | 否            | △ 能自动回填,但前端存 key 有 XSS 风险,需显式提示；谨慎             |
| 轻后端代理                    | 自建函数代理模型 API                         | **有** | **是**        | **✗ 本期**：破坏 INV-8 / 无后端身份                                |
| **Prompt Builder（推荐 v1）** | 把当前站点状态整理成高质量 prompt + 回填说明 | 无     | 否            | **✓✓** 唯一不自毁、且贴合所有者工作流                              |

**结论**：v1 做 **Prompt Builder**（含「参考图 → 风格变量」的 prompt 版本,输出可粘回 `StyleEditor` 的 JSON）；成熟后再评估「用户自带 key」；**后端代理不在本模板的身份内**。

---

## 7. 架构红线（任何功能必须服从的不变量）

- **纯静态、无长期后端**；未配置即优雅降级（INV-8 / `check:dist`）。
- **单一事实源**：Wizard / AI 只能写 overrides + style config,**复用 `export.js` 序列化器,不新增第二份**。
- **凭据绝不入码**：API key / GitHub token 不进任何导出、提交、源文件（延续 `SPEC-3-PERSIST.md` §6）。
- **性能预算**：动效/组件默认只启 **Safe 档**,不得砸掉移动端 Lighthouse 基线（性能 98 / a11y 100）。

---

## 8. 主要风险

- **范围太大**：提案 ≈ 一个完整 AI 建站产品。MVP 必须收敛（§9）。
- **分发未解决**：没人用的生成器是爱好；先用现有模板验证需求,再投 Wizard。
- **AI 文案同质化**：薄 prompt → 通用文案（"我拍关于记忆和光的短片"）→ 把站点做成样板,正是要避免的失败模式。Prompt Builder 要**逼出用户自己的声音**（问真实经历、喂真实 bio）,不是凭空生成。
- **动效破坏性能/统一性**：分级 **Safe（CSS transition/reveal/hover）/ Medium（小范围 GSAP）/ Heavy（Three.js/particles）**,默认仅 Safe。做**受控 variant**,而非「组件市场」——你模板的价值就是统一克制,开放任意拼装是最快毁掉它的方式。
- **参考图 = 抄袭风险**：功能定位「**提取设计变量**」而非「复刻页面」；输出风格倾向描述,不输出「完全复刻此站」。

## 9. 优先级（替代提案的 P0–P9）

| 级别   | 项目                                 | 依赖 AI?   | 破坏静态? | 理由                                                        |
| ------ | ------------------------------------ | ---------- | --------- | ----------------------------------------------------------- |
| **P0** | 模板解耦 + 空载实例化                | 否         | 否        | 一切的前置（§3C）；没它 Wizard 无从谈起                     |
| **P1** | Goal Picker → preset                 | 否         | 否        | 全案最强一步,零 AI 依赖,直接复用 `STYLE_PRESETS` + 模块系统 |
| **P2** | 部署前审计（Layout Audit）           | 否         | 否        | 最硬的护城河,且最便宜——复用现有 validators（§10）           |
| **P3** | Prompt Builder（含参考图 prompt 版） | 是（外部） | 否        | 唯一不自毁的 AI 接入；贴合所有者工作流                      |
| **P4** | 文字版截图反馈 / 受控 variant 切换   | 部分       | 否        | 进阶增强,放手动模式                                         |
| **P5** | 真正 API 接入（用户自带 key）        | 是         | 否        | 远期；需明确 key 安全说明                                   |

**放行门槛**：先穿过 **P0–P2**、并找到 **≥5 个真实用户**,再用真实反馈决定是否投 P3+。首轮 5 人反馈已经取得，但它首先支持 UX 简化与模板差异化，并不自动支持 AI 投入。截图坐标定位、站内视觉提取、组件市场、云部署自动化——**后面或永不**。

当前状态：P0–P2 与反馈驱动的 UX1 已完成；下一门槛是任务制复测，而非继续堆功能，因此 P3 Prompt Builder 仍不启动。

## 12. 首轮反馈后的产品判断

- **用户不是缺旋钮，而是缺路径。** StyleEditor 不再把 10 个维度做成平级孤岛；预设、基础调参、情绪板成为三个连续工作区。
- **实时预览必须常驻。** 桌面左右分屏，移动端上下分屏；参数调整与完整网站结果同时可见。
- **“更多风格”必须包含结构差异。** Minimal Portfolio、Personal Journal、Gradient Studio 会改变模块集、导航、顺序、layout 和 Landing，而不只是 token。
- **就地编辑是辅助入口，不替代完整编辑器。** About、Works 等区块提供 Content/Style 快捷入口；首页保留唯一全局入口，避免移动端重复。
- **首页是模板可信度的第一关。** 默认首页改为真实 portrait 全屏构图，并为 journal / gradient 提供独立构图；后续复测重点应是首屏理解度与 CTA 点击。

> **一句话**：别把它当 AI 建站平台来做；当「给你这套有审美的静态模板,加一层 AI 辅助填充 + 部署前护栏」。最该先做的,恰恰是最不性感的那步——把模板和示例数据解耦,让别人能用。
